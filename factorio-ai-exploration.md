# Factorio × Claude Code: 着想から構想まで

## きっかけ: Factorio はコードで操作できるのか？

Factorio には公式の REST API は存在しないが、以下の2つのプログラマブルなインターフェースがある。

**Lua Modding API** がゲームの中核的な拡張手段。`data` ステージ（プロトタイプ定義）と `control` ステージ（ランタイムロジック）の2層構造で、エンティティの生成・破壊、レシピ変更、イベントハンドリングなどゲーム内のほぼすべてを制御できる。

**RCON（Remote Console）** は外部プログラムとの接続点。ヘッドレスサーバー起動時に `--rcon-port` と `--rcon-password` を指定すれば、TCP 経由で任意のコンソールコマンドを送信可能。`/c` プレフィックスで Lua コマンドも実行でき、Python や Node.js からの制御が現実的に行える。

Web API が欲しい場合は RCON をラップするブリッジを自作する必要があるが、`factorio-rcon` 系のライブラリが GitHub にいくつか存在する。

## 発展: Claude Code にプレイさせるのはどうか？

### 実現可能性の評価

技術的には可能だが、いくつかの本質的な課題がある。

**観測の問題が最大の壁。** Claude Code は画面を見ることができない。ゲーム状態をテキスト/JSON として取得する「観測 mod」を Lua で書き、`find_entities_filtered()` や `game.player.get_inventory()` などで周囲の情報をシリアライズして渡す必要がある。

**アクションの抽象化。** 人間のマウスクリック操作を `game.player.insert()`, `game.player.build()` 等の Lua コマンドに変換するレイヤーが必要。建設・移動・研究指示などを高レベルコマンドとして mod 側に実装するのが現実的。

**リアルタイム性とのギャップ。** Factorio はリアルタイム進行するため、Claude Code の応答待ち中もゲームが動き続ける。`game.speed = 0.1` での減速か、毎 tick 一時停止するターン制化 mod で対処する。

### 現実的なアーキテクチャ

```
Claude Code → Python ブリッジ → RCON (TCP) → Factorio ヘッドレスサーバー
```

このパイプラインで、観測データの取得とコマンドの発行を交互に繰り返す。

### 難易度の段階的評価

- **簡単な自動化**（鉄板を作り続ける等）: すぐに実現可能
- **序盤の自動プレイ**（採掘→製錬→自動化ライン構築）: 観測/アクション mod をしっかり作れば可能
- **ロケット打ち上げまで完全自律**: 研究プロジェクト級の作り込みが必要

## さらなる発展: マルチエージェント化

### なぜ Factorio はマルチエージェント向きなのか

Factorio のヘッドレスサーバーは複数プレイヤーの同時接続を前提に設計されており、Lua 側で `game.players` から各プレイヤーを個別に操作できる。1つの RCON 接続から全プレイヤーを制御可能という特性がある。

加えて、Factorio は空間的に分業しやすいゲーム構造を持つ（採掘エリアと工場エリアが物理的に分かれる等）。これはマルチエージェント設計と自然に噛み合う。

### 制御モデルの選択肢

**中央集権型:** 1つの Claude Code セッションが全エージェントを統括。実装がシンプルでエージェント間の衝突を回避しやすいが、コンテキストが膨大になる。

**分散型:** 各 Claude Code インスタンスが担当エージェントの観測のみを受け取り自律判断。共有状態の同期（何が足りない、何を建設中か等）が課題だが、`game.write_file()` や共有ログで連携可能。

### プレイヤー実体の問題

通常はクライアント接続でプレイヤーが生成されるため、ダミープレイヤーの扱いが課題になる。より現実的なのは **プレイヤーは1人のまま、mod 側で複数の「仮想エージェント」を管理する** アプローチ。実際のプレイヤーエンティティではなくロボット的なエンティティを使い、各エージェントの行動をシミュレーションする。

## 構想: システム全体設計

### アーキテクチャ図

```
┌─────────────────────────────────────────────────┐
│                  Claude Code                     │
│  ┌───────────┐ ┌───────────┐ ┌───────────┐      │
│  │ Planner   │ │ Miner     │ │ Builder   │ ...  │
│  │ (総指揮)  │ │ (採掘)    │ │ (建設)    │      │
│  └─────┬─────┘ └─────┬─────┘ └─────┬─────┘      │
│        └──────────────┼──────────────┘           │
│                       ▼                          │
│              共有ゲーム状態 (JSON)                │
└───────────────────────┬─────────────────────────┘
                        │ RCON (TCP)
                        ▼
┌───────────────────────────────────────────────────┐
│           Python ブリッジ                          │
│  - RCON 通信・コマンドキュー管理                    │
│  - 観測データのシリアライズ                         │
│  - ゲーム速度制御                                  │
└───────────────────────┬───────────────────────────┘
                        │ RCON
                        ▼
┌───────────────────────────────────────────────────┐
│       Factorio ヘッドレスサーバー                   │
│  ┌──────────────────┬──────────────────┐          │
│  │    観測 Mod      │  アクション Mod   │          │
│  │ エンティティ探索  │  建設/撤去        │          │
│  │ インベントリ取得  │  移動/テレポ      │          │
│  │ マップスキャン    │  研究指示         │          │
│  │ 生産統計         │  アイテム操作      │          │
│  └──────────────────┴──────────────────┘          │
└───────────────────────────────────────────────────┘
```

### エージェント役割分担

| エージェント | 役割 | 主な操作 |
|---|---|---|
| **Planner** | 全体計画・優先度決定 | 研究ツリー解析、生産目標設定 |
| **Miner** | 資源採掘・輸送 | 採掘機配置、ベルト敷設 |
| **Builder** | 工場レイアウト | 組立機・製錬炉配置、電力網構築 |
| **Logistics** | 物流最適化 | ロジスティクスネットワーク、列車網 |
| **Defense** | 防衛 | 壁・タレット配置、弾薬補給 |

### 意思決定ループ

```
毎ゲームサイクル（数秒〜数十秒ごと）:
  1. [観測]   各エージェントの担当領域のゲーム状態を取得
  2. [評価]   Planner が全体の優先度を判定
              資源不足？ → Miner に採掘拡張を指示
              ボトルネック？ → Builder にライン増設を指示
              バイター接近？ → Defense に防衛強化を指示
  3. [行動]   各エージェントが割り当てられたタスクを実行
  4. [記録]   行動結果をログに書き出し、次サイクルの入力にする
```

### 観測データ構造

```json
{
  "tick": 12345,
  "player": {
    "position": { "x": 0, "y": 0 },
    "inventory": { "iron-plate": 50, "copper-plate": 30 }
  },
  "nearby_entities": [
    { "name": "iron-ore", "position": { "x": 5, "y": 3 }, "amount": 1200 }
  ],
  "production_stats": {
    "iron-plate": { "input_per_min": 30, "output_per_min": 25 }
  },
  "research": {
    "current": "automation-2",
    "progress": 0.45
  },
  "alerts": [
    { "type": "enemy_approaching", "position": { "x": 100, "y": -50 } }
  ]
}
```

### アクションコマンド体系

```lua
/agent miner build_mining_outpost iron {x=100, y=50}
/agent builder place_smelting_array iron 12
/agent builder connect_belt {x=10,y=5} {x=30,y=5}
/agent logistics setup_train_route "iron_outpost" "main_base"
/agent defense build_wall_perimeter {x=-50,y=-50} {x=150,y=100}
/agent planner set_research "logistics-2"
```

効率的な既知レイアウト（製錬配列、バランサー等）は mod 内にブループリント文字列として保持し、コマンド一つで設置可能にする。

### ゲーム速度制御戦略

| 方式 | 仕組み | 適用場面 |
|---|---|---|
| **ターン制** | 毎 N tick で一時停止、判断後に再開 | 開発・デバッグ時 |
| **低速モード** | `game.speed = 0.1` | 連続プレイのデモ |
| **イベント駆動** | 重要イベント発生時のみ停止 | 効率重視の本番運用 |

## 開発ロードマップ

### Phase 0: 基盤構築
- ヘッドレスサーバーの起動・RCON 接続確認
- 最小限の観測 mod（プレイヤー位置・インベントリ取得）
- Python ブリッジの RCON 通信部分
- Claude Code → ブリッジ → サーバーの疎通確認

### Phase 1: 単体エージェント
- 1体で「鉄鉱石を掘って製錬する」を達成
- 観測→判断→行動ループの安定動作
- 基本的なエラーハンドリング（建設失敗、資源不足等）

### Phase 2: 自動化ライン構築
- 赤サイエンスパックの自動生産まで到達
- ブループリントシステムの統合
- 電力網の自動管理

### Phase 3: マルチエージェント化
- 仮想エージェントの分離と役割分担
- Planner によるタスク割り当てロジック
- エージェント間の競合回避（同じ場所に建設しない等）

### Phase 4: 中盤攻略
- 列車網の自動構築
- ロジスティクスロボット活用
- バイター防衛の自動化
- 研究ツリーの戦略的選択

### Phase 5: ロケット打ち上げ
- サイエンスパック全種の自動生産
- サテライト製造・ロケット組立
- 完全自律でのロケット打ち上げ達成

## リスクと対策

| リスク | 影響 | 対策 |
|---|---|---|
| コンテキスト長の爆発 | 観測データが Claude Code の制限に達する | 抽象度を動的に調整、詳細は必要な領域のみ取得 |
| RCON レイテンシ | 判断→実行の遅延で状況が変化 | ターン制 or 低速モードで緩和 |
| 空間推論の限界 | LLM はベルト配置最適化が苦手な可能性 | ブループリントのテンプレートで補完 |
| エラーの連鎖 | 1つの判断ミスが計画全体を崩壊させる | チェックポイント保存、ロールバック機構 |
| mod の複雑化 | 観測/アクション mod のメンテコスト増大 | コマンド体系を最小限に保ち段階的に拡張 |

## 面白い発展の可能性

**人間との協力プレイ。** 人間がマップの一部を担当し、Claude エージェント群が残りを運営する。人間が戦略を指示し、エージェントが実行するスタイルも可能。

**複数インスタンスの競争。** 別々のマップで同時にプレイさせ、工場効率やロケット到達時間を比較する。プロンプトや戦略の違いがどう結果に影響するかの実験にもなる。

**自己改善ループ。** プレイログを分析し、ボトルネックや非効率だった判断を特定して、次回のプレイで戦略を改善する。

**別 Claude による実況解説。** プレイ中のゲーム状態を別の Claude インスタンスに渡し、リアルタイムで実況・解説させる。デモとしてのインパクトが大きい。

## 先行研究との関連

Minecraft では LLM にゲームをプレイさせる研究が活発で、CraftJarvis や Voyager などの事例がある。Factorio は Minecraft と比べて以下の点で異なる特性を持つ。

- **計画性の重要度が高い**: 生産チェーンの設計は長期的な見通しが必要で、LLM の推論能力が活きる
- **定量的な最適化**: スループット計算やレシオ設計は数値的な推論を要求する
- **空間制約**: ベルトや配管のルーティングは空間推論能力が試される（LLM が苦手な可能性あり）
- **段階的な複雑さ**: 序盤は単純、終盤は複雑という自然なカリキュラムが存在する

これらの特性から、Factorio は LLM の計画・推論能力のベンチマークとしても興味深い対象と言える。
