## この連載で何を目指すのか？

目標は2段階です。

- MVP: Nauvis軌道に到達して、宇宙側で運用を開始する
- 最終目標: Space Age の勝利条件である『solar-system-edge』到達まで行く

「とりあえず自動化っぽく動いた」では終わらせません。  
勝利条件までを、再現可能な形で持っていくのが今回の本題です。

もう少し記事としての狙いを言うと、この連載のターゲットは次の2つです。

- AI にゲームをやらせるときの設計や実装に興味がある人
- Factorio を題材に、自律エージェントやゲーム自動化の技術ノウハウを見たい人

なので、進捗報告だけでなく、なるべく「何を考えてその実装にしたのか」まで書いていくつもりです。

## 何をやらないか

この企画では「ゲーム内で実際にできないこと」はやりません。

例えば、以下は明確に禁止です。

- エンティティがいない遠隔地点への瞬間設置
- アイテムの直接生成
- 未研究テクノロジーの先取り利用
- 作業時間の不自然な短縮

逆に、ゲーム内で成立している仕組みは使います。  
ロボットも機械も使う。ただし、行動主体は制約を受ける。これを崩しません。

## どう実現するのか？

構成はシンプルです。

- ローカルで起動する Factorio ヘッドレスサーバー
- **fwai_bridge** という自作Mod。観測と行動の橋渡しを担当する部分
- Controller。Codex が操作する外部プロセス

ポイントは、AIの行動主体をゲーム内で見える存在にすることです。  
いまは **fwai-bot** を『spider-vehicle』ベースで作る方針で進めています。  
「見えない神の手で工場が伸びる」状態は避けます。

### 使っている技術

ここはもう少し具体的に書いておきます。

まずゲーム側は、Factorio の Lua Modding API を使っています。  
**fwai_bridge** という自作Modの中で、『data.lua』では **fwai-bot** を定義し、『control.lua』では観測と行動のためのコマンドを公開しています。  
現時点では『/fwai.health』『/fwai.observe』『/fwai.act』『/fwai.bot_state』『/fwai.reset』あたりが生えている状態です。

外からの制御には RCON を使っています。  
Controller は Python で書いていて、RCON パケットを直接しゃべる薄いクライアントを自前実装しています。  
やっていることはシンプルで、JSON を base64 にして『/fwai.observe』や『/fwai.act』に投げ、返ってきた JSON をまた Python 側で解釈する、という形です。

観測データとしては、少なくとも次を取れるようにしています。

- プレイヤー位置
- Bot の位置と状態
- 手持ちインベントリ
- 周辺エンティティ
- 現在の研究状況

アクション側はまだ最小構成で、今ちゃんと通っているのは『spawn_bot』『move』『wait』くらいです。  
逆に言うと、採掘や設置はこれからです。

### 実行環境はどうしているのか？

サーバーは、このリポジトリからローカル起動しています。  
Steam 版の Factorio バイナリをそのまま使いつつ、『--config』と『--mod-directory』をサーバー専用に分けて、『.runtime/server/』配下に閉じ込めています。  
これをやっている理由は、普段遊ぶクライアント環境とぶつけたくないからです。

さらに macOS では『launchd』から起動できるようにしていて、バックグラウンドでヘッドレスサーバーを上げられるようにしています。  
RCON は『27015/TCP』、ゲームポートは『34197/UDP』で待ち受けています。

クライアント側の Mod 同期も、Mod ポータル前提にはしていません。  
**fwai_bridge** はローカルの Mod ディレクトリに symlink で差し込む方式にしていて、リポジトリ側の変更をそのままプレイヤー側にも反映できるようにしています。

### いまの全体像

いまの構成を雑に書くと、こんな感じです。

```text
Codex
  -> Python Controller
  -> RCON
  -> Factorio headless server
  -> fwai_bridge
  -> visible bot (fwai-bot)
```

この構成にしている理由は、ゲームの画面そのものを AI に読ませるのではなく、  
「観測は Mod から構造化データで出す」「行動はゲーム内で成立する単位に落として返す」を徹底したかったからです。

## いまどこまで進んだか？

2026-03-06時点で、基盤は動き始めています。

![Factorio上に表示されたFWAI Bot。プレイヤーと同じワールド内に存在している](./images/factorio-window-20260306-092635.png)

実際に、プレイヤーとは別に FWAI Bot がゲーム内へ出ているところまでは確認できました。

- Space Age前提のローカルサーバーを起動済み。launchd 運用にも対応
- LAN参加で人間クライアントから普通に観測可能
- Mod同期問題に対応。ローカルクライアント側へ **fwai_bridge** を自動反映
- ControllerとRCONの最小ループを実装。health と observe と act の骨格まで通した

### 何が実際に動いているのか？

ここも誤解がないように書いておくと、まだ「自律プレイが始まっている」わけではありません。  
いま動いているのは、主に基盤です。

- ヘッドレスサーバーが安定して起動する
- 人間クライアントが LAN 参加できる
- **fwai-bot** がゲーム内に可視の実体として出る
- Controller から RCON 経由で health check と observe ができる
- 最小の action として Bot の spawn と move が通る

つまり、ゲームを進めるための手足そのものより、  
まず「どの制約で、どう観測し、どう命令を渡すか」のレイヤーを固めている段階です。

### まだできていないこと

逆に、まだ出来ていないこともはっきりしています。

- 手掘りの本実装
- 距離制約つきの設置
- インベントリ増減を伴う搬送
- 序盤の組み立てライン構築

このへんが出来て初めて、「見えている Bot が、ルールを守って実際に進捗を作る」と言えるようになります。

つまり「接続して観測する」までは通りました。  
次は「制約を守りながら実際に進捗を作る」フェーズです。

## 次回からやること

次回は、序盤の実作業に直結するところを進めます。

- 『move』だけでなく『mine』『place』『insert』を実装
- 距離・所持・衝突・研究などの制約チェックを厳格化
- Botが序盤を自力で立ち上げる最小シナリオを作る
